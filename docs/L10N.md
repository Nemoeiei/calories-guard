## Why hand-rolled instead of gen-l10n?

The standard `flutter gen-l10n` + `.arb` pipeline adds build-time codegen and
a `l10n.yaml`. That's the right call once we have hundreds of strings, ICU
plurals, or a separate translator workflow. We don't have any of those yet.

What we needed for closed beta: **a non-Thai tester can sign up, log in, and
record a meal without seeing Thai.** That's ~35 strings. A
`Map<String, Map<String, String>>` gets us there without a new build step and
stays reviewable in a single file.

When the catalogue crosses ~80 keys, or the product adds a third locale,
migrate to `.arb`. The public API (`AppLocalizations.of(context).tr('key')`,
`AppLocalizations.delegate`, `supportedLocales`) is compatible with what
gen-l10n emits, so call sites don't change.

## Usage

```dart
final l10n = AppLocalizations.of(context);
Text(l10n.tr('login.title'));
```

Missing-key behaviour (intentional):
1. Look up the requested locale's catalogue.
2. Fall back to English.
3. Fall back to the key itself — never throws.

A missing translation should never crash a production screen; the test file
has a `catalogue symmetry` check that fails CI if an English key lacks a Thai
counterpart (or vice versa). Add new keys to **both** language maps in the
same commit.

## Scope — what's translated today

| Screen | Keys |
|---|---|
| Welcome | `welcome.*` |
| Login | `login.title`, `login.email`, `login.password`, `login.forgot`, `login.cta` |
| Register (title + primary CTA only) | `register.*` |
| Record food (headers + meal type chips) | `record.*` |
| Home dashboard (KPI labels) | `home.*` |
| API error snackbars | `error.*` |

Validator-level micro-copy ("please enter email", "invalid format") is
**not** translated in this pass. It's a long tail of strings inside form
validators, and a non-Thai user still gets the field hint in English, which
is enough signal for beta. Tackle those during the gen-l10n migration.

## Device locale resolution

Defined in `main.dart` → `localeResolutionCallback`:

1. If the device locale matches a supported language code (`th` or `en`), use
   it.
2. Otherwise default to Thai (our primary market). A French-only tester
   still sees Thai UI chrome but English error messages via the fallback
   chain above — imperfect, but better than a mix of two random languages.

---

## When you add a new string

1. Add the key to **both** `'th'` and `'en'` maps in
   `app_localizations.dart`.
2. Add the key to the `keys` list in `l10n_test.dart` (the symmetry check).
3. Use `AppLocalizations.of(context).tr('your.key')` at the call site.

If step 2 is missed, the symmetry test fails and CI blocks the PR — which is
exactly the point.
