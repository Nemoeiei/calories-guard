# AI Coach Architecture (Calorie Guard)

ระบบ "Chatbot Coaching" หรือ "AI Coach" ที่เราสร้างขึ้นใน Calorie Guard เป็นสถาปัตยกรรมแบบ **Hybrid Agentic AI** ที่ผสมผสานความฉลาดสองรูปแบบเข้าด้วยกัน เพื่อให้คำแนะนำเรื่องโภชนาการและการออกกำลังกายที่แม่นยำที่สุด

## 1. ภาพรวมของสถาปัตยกรรม (Architecture Overview)

สถาปัตยกรรมประกอบด้วย 3 องค์ประกอบหลัก:
1. **Data Context Retrieval (ฐานข้อมูล)**: ดึงข้อมูลโปรไฟล์ผู้ใช้, ประวัติน้ำหนักย้อนหลัง, ประวัติการกิน และข้อมูลที่แพ้
2. **Machine Learning / Analytics Engine (สมองซีกซ้าย)**: สกัดข้อมูลตัวเลข หาความถี่ สัดส่วน และทิศทาง (Trends)
3. **LLM Orchestrator (สมองซีกขวา - Gemini)**: เข้ามาเพื่อตีความข้อมูลเหล่านั้น และสร้างคำพูดที่เป็นธรรมชาติ ส่งกำลังใจ และให้คำปรึกษาที่ตรงจุด

---

## 2. การทำงานของ Machine Learning / Analytics Engine 

เราเลือกใช้ ML เบื้องต้นและ Data Analytics แทน Deep Learning ล้วน ๆ เนื่องจากมีความแม่นยำและเสถียรสำหรับข้อมูลที่ไม่ต่อเป็นลำดับเวลาชัดเจน (Sparse Data)

### 2.1 Weight Trend Analyzer (Linear Regression)
* **โมเดลที่ใช้**: `sklearn.linear_model.LinearRegression`
* **การทำงาน**: โมเดลจะรับข้อมูลน้ำหนักรายวันย้อนหลัง (อย่างน้อย 3 วัน) สร้างเส้นแนวโน้ม (Trend Line)
* **ผลลัพธ์**: คำนวณความชัน (Slope) ถ้าความชันติดลบแสดงว่าน้ำหนักกำลังลด และคำนวณวันที่จะถึงเป้าหมาย (`days_estimated`) เพื่อนำไปเป็นตัวเลขเชิงจิตวิทยาให้ Gemini บอกผู้ใช้

### 2.2 Food Analyzer (Macronutrient Gap & Frequency)
* **การทำงาน 1 (Nutrition Gap)**: เทียบเป้าหมายโปรตีนและแคลอรี่ที่ระบบตั้งไว้ กับสิ่งที่ผู้ใช้กินเฉลี่ย 7 วันย้อนหลัง เพื่อวิเคราะห์หาจุดบกพร่องที่ร้ายแรง (Critical Issues) เช่น "โปรตีนต่ำเกินไป", "แคลอรี่สูงกว่าเป้าหมายมาก"
* **การทำงาน 2 (Frequent Foods)**: ใช้ Pandas เพื่อสรุปความถี่ (Frequency) หาอาหาร 5 อันดับแรกที่ผู้ใช้กินบ่อยที่สุด เพื่อให้ LLM นำไปแนะนำเมนูที่ลดแคลอรี่แต่ยังตรงกับความชอบ

---

## 3. การทำงานของ Agentic AI (LLM Orchestrator)

เมื่อ ML ทำงานเสร็จสิ้น โค้ดใน `chatbot_agent.py` จะทำหน้าที่เป็น Orchestrator:
1. รวบรวมข้อมูลทั้งหมด (Profile, Allergies, ML Trends, Gaps)
2. สร้าง **Context-Aware Prompt** ซึ่งมีโครงสร้างกำหนดให้ Gemini สมบทบาทเป็น "โค้ชแคลเซียม"
3. ส่งมอบข้อมูล Insights เป็น Context (Background System)
4. ส่งคำถามล่าสุดของผู้ใช้ เข้าไปยัง API
5. คืนค่าผลลัพธ์เป็นภาษาไทยที่สุภาพ เข้าใจง่าย และจับต้องได้

## 4. API Endpoints

ผู้ดูแลแอป/Frontend สามารถเรียกใช้งานได้ผ่าน:

**POST** `/api/chat/coach`
```json
// Request
{
  "user_id": 1,
  "message": "สามวันนี้น้ำหนักไม่ลงเลย เกิดอะไรขึ้น?"
}

// Response
{
  "response": "สวัสดีครับโค้ชแคลเซียมมาแล้ว! จากที่โค้ชดูข้อมูลย้อนหลัง 3 วัน แม้ว่าแนวโน้มน้ำหนักจะคงที่ แต่ระดับโปรตีนของคุณค่อนข้างขาดหายไปเฉลี่ย 30g นะครับ ทำให้ระบบเผาผลาญอาจจะชะลอตัว โค้ชจำได้ว่าคุณชอบกินกะเพราไก่ ลองเปลี่ยนเป็นกะเพราอกไก่ดูไหมครับ..."
}
```

## 5. ทำไมไม่ใช้ LSTM เต็มรูปแบบ?
ระบบประเมินว่า LSTM ที่เรียนรู้ Time-Series เหมาะแก่งานที่มีข้อมูลเรียงต่อกัน 100% ไม่มีช่วงฟันหลอ การใช้ Analytics + ML Base Model ทำให้ลดต้นทุนเซิร์ฟเวอร์, อ่านเหตุผลให้ LLM พิมพ์อ้างอิงง่ายกว่า (Explainable AI) และให้ผลลัพธ์ที่ตอบโจทย์ Recommender System ในบริบทนี้ดีกว่าครับ
